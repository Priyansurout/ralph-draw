## Codebase Patterns

### Package.json Structure
All package.json files use `"type": "module"` to enable ES module imports. Sub-packages omit `private: true` (root only). Caret (`^`) version specifiers used throughout. `socket.io-client` belongs in `dependencies` (runtime), not `devDependencies`. `react` and `react-dom` must share identical version strings.

### ES Module Compatibility
`nanoid ^5` is pure ESM — must use `^5` (not v3/v4 which are CJS). Server and client socket.io packages must share the same major version to ensure protocol compatibility.

### Server Entry Point Pattern
`server/index.js` uses `createServer(app)` from Node's `http` built-in (NOT `app.listen()`) so Socket.IO and Express share the same port. `new Server(httpServer, { cors: { origin: 'http://localhost:5173' } })` restricts CORS to the Vite dev origin. `roomManager` is created at module scope (not inside connection handler) so all socket handlers close over it. Relative imports must use `.js` extension: `import { createRoomManager } from './rooms.js'`.

### Room Manager Pattern
`createRoomManager()` is a factory function returning a plain object. Two Maps: `rooms` (roomId→{users:Map,chatHistory:[]}) and `socketToRoom` (socketId→roomId). `removeUser` deletes the `socketToRoom` entry first before checking roomId (defensive against double-disconnect). Empty rooms are deleted when last user leaves. `getUsersArray` returns `Array.from(room.users.values())` — a fresh snapshot array.
